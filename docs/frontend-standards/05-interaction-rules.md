# 05 交互规则

交互应优先稳定、清晰、可预测。后台系统的核心体验是反复查询、筛选、编辑、审核和导出，因此不要为了视觉效果牺牲效率。

## 查询与筛选

查询流程：

1. 用户输入筛选条件。
2. 点击“查询”。
3. 表格 loading。
4. 返回数据、空状态或错误提示。

规则：

- “查询”按钮使用主按钮。
- “重置”按钮恢复默认筛选条件，并可立即重新查询。
- 低频字段进入“更多字段”。
- 展开更多字段后，不应造成查询按钮不可见。
- 筛选条件过多时，优先使用 Popover、Drawer 或多行布局。

## 表格选择

依赖选中行的按钮：

- 未选中时应禁用，或点击后给出明确提示。
- 单选操作应限制只能选择一行。
- 批量操作应显示已选数量。

危险批量操作必须二次确认。

## 新增、编辑、详情

推荐模式：

- 新增：Drawer 或独立表单页。
- 编辑：Drawer 优先，复杂表单用独立页面。
- 查看详情：Drawer 或详情页。

详情页信息应分区展示，不要把所有字段堆成单列长表。

## 删除与危险操作

危险操作包括：

- 删除
- 驳回
- 撤回
- 作废
- 停用
- 清空

规则：

- 必须有二次确认。
- 危险按钮使用错误色或 danger。
- 确认文案应说明影响对象和后果。
- 成功后刷新当前数据，不跳到无关页面。

## 加载态

需要 loading 的场景：

- 页面首次加载。
- 表格查询。
- 提交表单。
- 导入导出。
- 同步数据。

规则：

- 查询 loading 应绑定表格区域，不遮挡整个系统。
- 提交 loading 应绑定按钮，防止重复提交。
- 超过 3 秒的操作应给出明确文案。

## 空状态

空状态应说明下一步：

- 查询无结果：提示“暂无符合条件的数据”，可引导重置筛选。
- 新模块无数据：提示可新建或导入。
- 权限无数据：提示联系管理员或检查权限。

禁止只显示空白表格。

## 错误态

错误提示应包含：

- 发生了什么。
- 用户能做什么。
- 必要时显示错误码或请求信息。

表单错误信息紧贴字段下方。接口错误使用统一消息提示或页面内错误区。

## 反馈消息

- 成功：短 toast。
- 失败：明确错误 toast 或弹层。
- 警告：用于可能影响结果但不阻断的情况。
- 导入导出：长耗时操作应有进度或结果反馈。

文案使用用户能理解的业务语言，不暴露内部变量名。
